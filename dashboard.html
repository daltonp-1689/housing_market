Where This Is All Going

Every small step you take feeds into a larger arc. Here is the full journey — keep this in view even on the earliest steps. You are not just learning individual tools; you are building a complete data science system from scratch.

1
Foundation — Version Control & Project Structure
Git, GitHub, and a clean project layout. Every professional data science project starts here. Without version control, you cannot collaborate, roll back mistakes, or share reproducible work.
2
Data Acquisition — Getting Real Housing Data
Sourcing, downloading, and storing raw data about El Dorado Hills and Folsom home sales. ML models are only as good as the data behind them — understanding where your data comes from is non-negotiable.
3–4
pandas & Jupyter — The Core Data Science Toolkit
pandas is the library that lets you manipulate tabular data in Python. Jupyter notebooks are the interactive environment where data scientists explore, visualize, and document their thinking. These two tools are used in virtually every real-world data science project.
5
EDA — Exploratory Data Analysis
Before building any model, you need to understand your data deeply — distributions, outliers, correlations, missing values. EDA is how you ask questions of your data before the data asks questions of you.
6
Data Pipeline — Automating the Flow from Raw to Ready
A pipeline takes raw data and transforms it into clean, feature-rich data ready for a model. This is where data cleaning, feature engineering, and preprocessing live. Building a reusable pipeline means you can re-run the whole process with one command.
7
Machine Learning — Predicting Home Prices
Train models that learn patterns in historical sales data and predict future prices. You will explore regression algorithms, evaluate model accuracy, tune parameters, and understand what features drive value in El Dorado Hills and Folsom specifically.
8
AI Integration — Claude-Powered Market Summaries
Use the Claude API to automatically generate plain-English market summaries from your model's predictions and data insights. This is where your data pipeline and ML output become something a real person can read, act on, and share.

Module 1 — GitHub & Project Setup

Before any data science can happen, you need a place to put your work and a system to track changes. This module establishes the professional foundation that every real project is built on.

Step 1 Create the Repository & .gitignore
Complete ▶
What Was Built
A Git repository was initialized for the project and pushed to GitHub. A .gitignore file was added to tell Git which files to never track — things like temporary Python files, local environment settings, and the data/ folder (since raw data files are often too large for GitHub).
Why It Matters
Every change you make from now on is tracked. You can always go back in time, compare versions, and share your work with others. This is the same workflow used by professional data scientists and engineers at every company.
What I Learned
Git is a version control system — a tool that takes snapshots of your project over time. GitHub is a website that stores those snapshots in the cloud. A .gitignore is a list of patterns telling Git "ignore these files entirely."

📁 housing_market/ 📄 .gitignore 📄 README.md
Key Commands Used
bash Terminal
# Initialize a new Git repository
git init

# Stage all files for the first commit
git add .

# Save a snapshot with a descriptive message
git commit -m "Initial commit"

# Connect to GitHub and push
git remote add origin <your-repo-url>
git push -u origin main
How This Connects to Machine Learning

Every ML project lives inside a Git repository. When you eventually train models, you will version not just your code but also your model configurations, training scripts, and results. The habits you build now — committing often, writing clear messages — are the same ones that prevent "which version of the model was this?" disasters later. The .gitignore pattern for data/ is standard in ML: large datasets go in separate storage (S3, a database, a data lake), not inside the repo.

Glossary — All Terms Introduced So Far

Every technical term introduced in this project is defined here in plain English. New terms are added automatically after each step.

Git
Module 1 · Step 1
A version control system — a tool that saves snapshots of your project over time. Every time you make a commit, Git records exactly what changed and when. This means you can always go back to any previous state and see the full history of your work.
repository
Module 1 · Step 1
A project folder that Git is tracking. Often called a "repo" for short. It contains all your project files plus a hidden .git/ folder where Git stores its history. Think of it as a project folder with a built-in time machine.
commit
Module 1 · Step 1
A saved snapshot of your project at a specific point in time. Each commit has a unique ID and a message describing what changed. Commits are the individual entries in your project's history. A good rule: commit small and often, with clear messages.
push
Module 1 · Step 1
The act of uploading your local commits to a remote repository (like GitHub). After you commit, Git knows about the change — but only on your machine. A push sends those commits to the cloud so they are backed up and visible to others.
pull
Module 1 · Step 1
The opposite of push — downloading commits from a remote repository to your local machine. If someone else pushes new changes to GitHub (or if you pushed from a different computer), you run git pull to bring your local copy up to date.
clone
Module 1 · Step 1
Copying a remote repository (from GitHub) onto your local machine for the first time. Unlike pull (which updates an existing local repo), clone creates a brand-new local copy including the full commit history. The command is git clone <url>.
.gitignore
Module 1 · Step 1
A plain text file in your repository that lists files and folders Git should completely ignore. Common entries include __pycache__/ (compiled Python), .env (secret API keys), and data/ (large raw data files). If a file is in .gitignore, it will never show up in git status and will never be committed.
HTML
Module 1 · Step 2
HyperText Markup Language — the standard language for structuring content on the web. An HTML file is just a text file that browsers know how to read and render as a visual page. No server or special software is needed to open a local HTML file — any browser will do.
version control
Module 1 · Step 1
A system for tracking changes to files over time. Git is the most popular version control system. With version control, you can see what changed, who changed it, when, and why. It is the difference between saving document_final_v3_REALLYFINAL.docx and having a proper history.
GitHub
Module 1 · Step 1
A website (and cloud service) that hosts Git repositories. GitHub stores your project's full history in the cloud, makes it easy to share code with others, and provides tools for collaboration like pull requests and issues. Git is the tool; GitHub is the hosting platform. They are related but not the same thing.
