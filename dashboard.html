Step 1 Your Data Source — Zillow Research
Complete ▶
What We Found
Zillow Research publishes free, public CSV datasets updated monthly. The one we use is called ZHVI — Zillow Home Value Index — filtered to single-family homes and condos at the median tier. It contains one row per zip code and hundreds of monthly columns going back to 2000.
Why It Matters
Choosing a data source is a real data science decision. Zillow ZHVI is a smoothed, seasonally adjusted index — not raw sale prices. That means less noise but also a slight lag. For our goal (spotting trends and predicting home values) it is exactly the right level of signal.
What I Learned
ZHVI (Zillow Home Value Index) — a dollar estimate of the median home value in an area, updated monthly. Unlike a single sale price, it smooths over noise to show the true trend. Seasonally adjusted means the index removes predictable fluctuations (e.g., prices always dip in winter) so the signal is cleaner.
Our Target Zip Codes
95762 — El Dorado Hills, CA 95630 — Folsom, CA
What the CSV Looks Like

How This Connects to Machine Learning

Before you can train any model, you need a target variable — the thing you are trying to predict. ZHVI gives us exactly that: a clean, monthly median home value per zip code. Later, in Module 7, you will use the historical pattern in these columns to train a model that forecasts future values. The data shape (one row per zip, one column per month) will directly inform how you structure your feature matrix.

Step 2 Write the Fetcher Script
Complete ▶
What Was Built
src/agents/fetcher.py — a Python script that downloads the full Zillow ZHVI CSV, filters it to just the El Dorado Hills and Folsom zip codes, and saves the result to data/raw/ with a date-stamped filename.
Why It Matters
Instead of downloading files by hand every month, the script does it for you — and saves the result with a timestamp so you always have a clean historical archive. This is the foundation of every automated data pipeline: reliable, repeatable, hands-free acquisition.
What I Learned
urllib.request — Python's built-in library for making HTTP requests (downloading files from URLs). No install needed. csv.DictReader — reads a CSV as a stream of dictionaries, one per row. pathlib.Path — a modern way to work with file paths that works on Mac, Windows, and Linux.
File Created
📄 src/agents/fetcher.py
The Fetcher — Core Logic
python src/agents/fetcher.py
import csv, io, urllib.request
from datetime import datetime
from pathlib import Path

TARGET_ZIPS = {"95762", "95630"}   # El Dorado Hills, Folsom

DATA_URL = (
    "https://files.zillowstatic.com/research/public_csvs/zhvi/"
    "Zip_zhvi_uc_sfrcondo_tier_0.33_0.67_sm_sa_month.csv"
)

def fetch():
    # 1. Download the full Zillow CSV into memory
    with urllib.request.urlopen(DATA_URL) as resp:
        text = resp.read().decode("utf-8")

    # 2. Parse it and keep only our two zip codes
    reader = csv.DictReader(io.StringIO(text))
    rows = [r for r in reader if r["RegionName"] in TARGET_ZIPS]

    # 3. Save with a date-stamped filename
    out = Path("data/raw") / f"zhvi_edh_folsom_{datetime.now():%Y%m%d}.csv"
    with open(out, "w") as f:
        writer = csv.DictWriter(f, fieldnames=reader.fieldnames)
        writer.writeheader()
        writer.writerows(rows)

    print(f"Saved {len(rows)} rows → {out}")
How This Connects to Machine Learning

The pattern here — download → filter → save to a known location — is called an ETL step (Extract, Transform, Load). Every ML pipeline starts with ETL. Right now our "transform" step is minimal (just filtering rows). In Module 6 you will build the full transformation — cleaning nulls, engineering features, normalizing values — on top of this same raw file. Keeping raw data untouched in data/raw/ means you can always re-run the pipeline from scratch.

Step 4 Schedule It — Your First Agent
Upcoming ▶
What We Will Do
The fetcher already has a --schedule flag built in. Running it with that flag starts an APScheduler loop that calls fetch() every 24 hours automatically — no cron job, no cloud service needed. The agent runs as long as the process is alive.
Why It Matters
This is the moment the script becomes an agent: code that acts autonomously on a schedule, not just when you run it by hand. The Fetcher agent is the first component of the multi-agent system you will complete in Module 9. Every fresh CSV it saves is a timestamped snapshot of the market — your historical archive.
What I Will Learn
APScheduler — a Python library that runs functions on a schedule (interval, cron, one-time). A BlockingScheduler keeps the process alive and runs jobs in the foreground. The pattern of "run once immediately, then on an interval" is standard in data agents so you always have fresh data without waiting for the first interval to elapse.
New Concept
Agent — code that runs autonomously on a trigger (a schedule, an event, or a call from another agent) rather than waiting for a human to click run. The Fetcher is the simplest possible agent: one job, one schedule, one output.

How APScheduler Works
python src/agents/fetcher.py — scheduler section
from apscheduler.schedulers.blocking import BlockingScheduler

def start_scheduler(interval_hours=24):
    scheduler = BlockingScheduler()

    # Register the fetch() function as a job
    scheduler.add_job(fetch, "interval", hours=interval_hours)

    fetch()            # run once right now
    scheduler.start()  # then keep running on the interval
How This Connects to Machine Learning

Production ML systems rely on scheduled agents to stay current. A model trained on last year's data is less useful than one retrained on data from yesterday. The Fetcher agent you are building here is the first piece of that system: it ensures your training data is always fresh. In Module 9, the Fetcher's output will automatically trigger the Analyst agent, which will trigger the Reporter — a fully autonomous loop from raw data to finished summary.

Glossary — All Terms Introduced So Far

Every technical term introduced in this project is defined here in plain English. New terms are added automatically after each step.

Git
Module 1 · Step 1
A version control system — a tool that saves snapshots of your project over time. Every time you make a commit, Git records exactly what changed and when. This means you can always go back to any previous state and see the full history of your work.
repository
Module 1 · Step 1
A project folder that Git is tracking. Often called a "repo" for short. It contains all your project files plus a hidden .git/ folder where Git stores its history. Think of it as a project folder with a built-in time machine.
commit
Module 1 · Step 1
A saved snapshot of your project at a specific point in time. Each commit has a unique ID and a message describing what changed. Commits are the individual entries in your project's history. A good rule: commit small and often, with clear messages.
push
Module 1 · Step 1
The act of uploading your local commits to a remote repository (like GitHub). After you commit, Git knows about the change — but only on your machine. A push sends those commits to the cloud so they are backed up and visible to others.
pull
Module 1 · Step 1
The opposite of push — downloading commits from a remote repository to your local machine. If someone else pushes new changes to GitHub (or if you pushed from a different computer), you run git pull to bring your local copy up to date.
clone
Module 1 · Step 1
Copying a remote repository (from GitHub) onto your local machine for the first time. Unlike pull (which updates an existing local repo), clone creates a brand-new local copy including the full commit history. The command is git clone <url>.
.gitignore
Module 1 · Step 1
A plain text file in your repository that lists files and folders Git should completely ignore. Common entries include __pycache__/ (compiled Python), .env (secret API keys), and data/ (large raw data files). If a file is in .gitignore, it will never show up in git status and will never be committed.
HTML
Module 1 · Step 2
HyperText Markup Language — the standard language for structuring content on the web. An HTML file is just a text file that browsers know how to read and render as a visual page. No server or special software is needed to open a local HTML file — any browser will do.
version control
Module 1 · Step 1
A system for tracking changes to files over time. Git is the most popular version control system. With version control, you can see what changed, who changed it, when, and why. It is the difference between saving document_final_v3_REALLYFINAL.docx and having a proper history.
GitHub
Module 1 · Step 1
A website (and cloud service) that hosts Git repositories. GitHub stores your project's full history in the cloud, makes it easy to share code with others, and provides tools for collaboration like pull requests and issues. Git is the tool; GitHub is the hosting platform. They are related but not the same thing.
skill (Claude Code)
Module 1 · Step 3
A slash command in Claude Code that gives Claude a specialized, reusable capability — like /xlsx for spreadsheets or /claude-api for building Anthropic API apps. Instead of re-explaining what you want each time, you invoke the skill and Claude follows a battle-tested playbook. You can also build your own custom skills using /skill-creator — which is exactly what Module 9 does to create a /housing-report command that generates a full market report in one shot.
virtual environment
Module 1 · Step 3
An isolated copy of Python dedicated to one project. When you activate a virtual environment (source .venv/bin/activate), any libraries you install go into that project's private folder, not your system Python. This prevents version conflicts between projects — Project A can use pandas 1.5 while Project B uses pandas 2.0 without either breaking.
requirements.txt
Module 1 · Step 3
A plain text file listing every Python library a project depends on, one per line. Running pip install -r requirements.txt installs all of them at once. This is how you share a reproducible environment — anyone who clones your repo can get the exact same setup with one command.
agent
Module 1 · Step 3
In AI, an agent is a program that perceives its environment, makes decisions, and takes actions — autonomously, without a human clicking "run" each time. In this project, agents are Python scripts that fetch data, run the ML pipeline, and generate reports on a schedule. The key difference from a regular script: an agent can react to what it finds, use tools, and hand work off to other agents.
APScheduler
Module 1 · Step 3
A Python library for scheduling code to run at specific times or intervals — daily, hourly, every Monday at 6am, etc. It is what will make this project's agents run hands-free. Think of it as a built-in alarm clock for your Python scripts.
